Attendance app — Step-by-step implementation guide

Overview
- Purpose: record attendance for students per session/subject and produce reports.
- Files: `models.py`, `forms.py`, `views.py`, `urls.py`, `admin.py`, `scripts/create_test_attendance.py`.

1) Models
- Example models: `AttendanceRecord`, `AttendanceSession`.

from django.db import models
from django.contrib.auth.models import User

class AttendanceSession(models.Model):
    date = models.DateField()
    subject = models.ForeignKey('courses.Subject', on_delete=models.CASCADE)
    teacher = models.ForeignKey(User, on_delete=models.SET_NULL, null=True)

class AttendanceRecord(models.Model):
    session = models.ForeignKey(AttendanceSession, related_name='records', on_delete=models.CASCADE)
    student = models.ForeignKey('accounts.StudentProfile', on_delete=models.CASCADE)
    status = models.CharField(max_length=10, choices=[('present','Present'),('absent','Absent'),('late','Late')])
    timestamp = models.DateTimeField(auto_now_add=True)

def mark_attendance(request, session_id):
    session = get_object_or_404(AttendanceSession, pk=session_id)
    if request.method == 'POST':
        # process submitted statuses for each student
        for key, value in request.POST.items():
            if key.startswith('status_'):
                student_id = key.split('_',1)[1]
                AttendanceRecord.objects.update_or_create(session=session, student_id=student_id, defaults={'status': value})
        return redirect('attendance_session_detail', session_id=session.id)
    students = session.subject.batch.studentprofile_set.all()
    return render(request, 'attendance/mark.html', {'session': session, 'students': students})

8) Notes
- Consider performance when marking many students: use bulk_create or bulk_update.
- Use permissions/decorators to allow only teachers/authorized staff to mark attendance.
 
Project-specific models (from this repository)

- `AttendanceSession`:
    - Key fields: `session_id` (auto-generated), `batch` (FK to `accounts.Batch`), optional `course`, `subject`, `teacher` (limit_choices_to role=teacher), `date`, `start_time`, `end_time`, `semester`, `year`, `total_students`, `present_count`, `absent_count`, `status`, `remarks`.
    - Methods: `clean()` ensures subject belongs to course and validates times; `save()` generates `session_id`; `update_stats()` recalculates counts; `attendance_percentage` property computes percentage.

- `AttendanceRecord`:
    - Key fields: `session` (FK to `AttendanceSession`), `student` (FK to `accounts.StudentProfile`), `date`, `status` (present/absent/excused/half_day), `check_in_time`, `recorded_by`.
    - Constraints: `unique_together = ('session', 'student', 'date')` and indexes for fast queries.
    - Behavior: `save()` and `delete()` call `session.update_stats()` to keep session statistics accurate.

Notes specific to this codebase
- Use `AttendanceSession.update_stats()` when bulk-updating records to avoid N+1 updates; prefer aggregations and a single update query.
- The `session_id` format uses batch code + date + sequence; respect this when creating sessions programmatically.
